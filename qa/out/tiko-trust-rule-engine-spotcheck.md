## Where the recall gap comes from

The 17 intra type-ref misses (0 supertype misses) break down into three families, all genuine jcma
recall gaps and all safe-degrading:

1. **Deeply-nested types (7 misses).** jcma resolves the outer nesting levels but misses the
   third-level nested member: `Action.CreateCase.Indicator` (`CreateCaseStrategy`, its test, and
   `ActionMapper`) and `Business.WeeklyHours` (`ServiceValidationEnrichService`).

2. **Types used only as a generic argument or nested-record field (7 misses).** `EvaluationRow`
   appears only inside `List<EvaluationRow>` in a nested `private record` (`OrchestratorService`
   and the three `Jooq*EvaluatorIT` tests); `RuleTriggeredEventMapper` / its nested
   `SupportedSchemaVersions` in `OrchestratorServiceIT` and the nested type's own self-reference.

3. **Scattered test-fixture references (3 misses).** Nested-entity / cross-package references in
   test code — `ConnectorJpa` + `Connector` (`CrudEventListenerIT`), the two `*EnrichedJpa`
   entities (`RafikiMergeEnrichServiceTest`), `Business` (`CrudEventListenerV2FlagEnabledIT`),
   `ActionTarget` (`TargetResolverTest`) — same nested/generic families as (1)/(2).

All misses are safe-degrading. There are **no** intra-repo extras (no silent-wrong dependencies). The
residual external extras (1.0%) are generic type-variable letters (`A`, `T`, `B`, `C`) and library
builder types — classpath/edge-case differences, not wrong repo dependencies.

## Oracle calibration (so the numbers measure resolution, not definition)

The javac oracle was extended during this exercise to count **every type-name position jcma's per-node
scan sees**, because each is a genuine type dependency that a naive oracle under-counts — making jcma
look falsely imprecise. Added: sealed **`permits`** entries, **class literals** (`Foo.class`, e.g.
MapStruct `@SubclassMapping(source = FooJpa.class)`), **qualified-name qualifiers** (`Outer` in
`Outer.Inner`), **switch type/record patterns** (`case Foo(int a) ->`), and **method-reference type
qualifiers** (`Type::method`). Before these, jcma's apparent intra precision was ~88%; every "extra"
removed by them was jcma being **correct** and the oracle incomplete. After calibration, intra
precision is 100% and the residual external extras (1.0%) are classpath/edge-case differences.

## Conclusion

On a foreign 207-type Maven repo, jcma's source-level dependency resolution is **exact where it
answers** (100% intra-repo / 99% external precision — no wrong dependencies) and **high-recall**
(supertypes 100%, type-refs 97.8% intra). The residual recall gap is entirely deep-nesting and
generic-argument edge cases — all safe-degrading, consistent with the PRD §4 contract.

## History — whole-file degradation (FIXED, commit `23d1106`)

The earlier run of this spot-check flagged one dominant, actionable finding: **`ActorServiceImpl`
whole-file resolver degradation** — the file parsed, but *every* type mention safe-degraded to
`UNRESOLVED`, so jcma emitted nothing for the whole compilation unit (~54% of the then recall gap;
the lone owner type jcma never emitted). Root cause: a `yield` (switch-expression arm) parsed at
`LanguageLevel.RAW` produced a parse problem, so JavaParser skipped `SymbolResolver` injection for
the entire unit (and RAW's recovery discarded the yield arm). Fixed by parsing at the project's
discovered Java level with the language-level validator stripped (docs/whole-file-resolution-
degradation.md). Post-fix this run: `ActorServiceImpl` emits 53 dependency rows with zero
divergences, the never-emitted-owner count is 0, and intra type-ref recall rose 95.3% → 97.8%.
