Hand analysis of the divergences the harness produces (the auto `…-report.md` and `…-diffs.tsv`,
both regenerated by `qa/scripts/qa-deps-accuracy.sh` and **not** committed), classifying each as a
real jcma miss, a real jcma extra, an oracle artifact, or an out-of-contract inferred type. This file
is written by hand and is the one committed QA artifact; re-run the script to refresh the numbers below.

> Updated after the **oracle partition** (TYPEREF vs TYPEREF_INFERRED): the oracle now splits the
> type names a program *writes in source* (jcma's navigation contract) from those javac only *infers*
> (`var` locals, implicit lambda params). The headline recall measures the former; inferred-only deps
> are reported as a separate, out-of-contract category. See "Why the partition" below.

## Genuine residual: none (deep-nesting fixed)

The only genuine residual used to be the **third-level nested record** `Action.CreateCase.Indicator`
(a `record Indicator` inside record `CreateCase` inside interface `Action`), written as a return type
in `ActionMapper` and `new Action.CreateCase.Indicator(…)` in `CreateCaseStrategyTest`. jcma resolved
the outer levels but safe-degraded on the descend-through-record hop.

Root cause was a JavaParser 3.28.2 bug: `JavaParserTypeAdapter.solveType` recurses into
class/interface/enum/annotation containers but not `record`s, throwing `UnsupportedOperationException`
for any type nested *through* a record. Fixed in jcma with a guarded, recursive **scope-nested
fallback** (`JavaParserEngine.resolveTypeRef` / `resolveViaScope`): when the stock resolve throws, it
resolves the scope (recursively — generalising to any nesting depth) and looks up the nested type by
exact simple name via the record-safe `internalTypes()`. Precision-preserving (emits only a type
confirmed inside an already-resolved scope), and on the shared `find_references` path so the
agent-facing tool benefits too. Post-fix: intra type-ref recall 99.7% → **100.0%**, precision held.

## Inferred-only type references (out of navigation contract): 15 intra

The former "recall gap" was dominated by types that **appear in the oracle only through an inferred
position** — a `var` local or an implicit lambda parameter, where the type name is never written in
source. jcma's syntactic `ClassOrInterfaceType` scan cannot see these by design, and neither does
IntelliJ's *Find Usages* on a type; the javac oracle is the stricter outlier (full compiler
attribution). The partition moves them out of the headline into their own category, where jcma's
coverage is 0% **by design** (not a bug). The 15 intra inferred-only deps:

- **Implicit lambda parameters** — `.map(row -> …)` → `EvaluationRow` (`OrchestratorService` + the
  three `Jooq*EvaluatorIT` tests); `.map(weeklyHours -> …)` → `Business.WeeklyHours`
  (`ServiceValidationEnrichService`); similar lambda-param inferences in the CRUD/merge IT fixtures
  (`ConnectorJpa`/`Connector`, the two `*EnrichedJpa` entities, `Business`).
- **`var` locals** — e.g. `final var targets = targetResolver.resolve(…)` → `ActionTarget`
  (`TargetResolverTest`); `var`-bound mapper/event types in `OrchestratorServiceIT`.

These are correct to exclude from the recall denominator: jcma's contract is "resolve what is written",
and an agent navigating "who uses `EvaluationRow`" via `find_references` gets the same answer IntelliJ
gives. (If inferred-site completeness ever becomes a requirement, it is a deliberate index change, not
a resolution bug — see the rejected resolve-file-only plan; deferred.)

## Why the partition (and the earlier precision calibration)

The javac oracle counts **every type-name position**, so it must be aligned with what each side can
*see*, or the numbers mislead:

- **Precision calibration (earlier):** the oracle was extended to count sealed **`permits`**, **class
  literals** (`Foo.class`), **qualified-name qualifiers** (`Outer` in `Outer.Inner`), **switch
  type/record patterns**, and **method-reference type qualifiers** — each a real written dependency a
  naive oracle under-counts, which had made jcma look falsely imprecise (~88% → 100% intra precision).
- **Recall partition (this exercise):** conversely, the oracle over-counts against a *navigation*
  tool when it credits inferred-only types (`var`/lambda) that no syntactic scanner can reach. Tagging
  those `TYPEREF_INFERRED` and reporting them out-of-contract makes the headline recall measure jcma's
  actual contract. Detection is source-faithful: a type position is "inferred" iff its source span is
  empty, the text `var`, or (for a lambda param) the parameter-name token.

## Conclusion

On a foreign 245-type Maven repo, jcma's source-level dependency resolution is **complete and exact on
its contract**: supertypes 100% / 100% and written type-refs **100% / 100%** intra-repo — every written
intra-repo type dependency javac sees is resolved, with no wrong edges. The former deep-nesting residual
(`Action.CreateCase.Indicator`) is fixed (recursive scope-nested fallback). The 15 inferred-only deps
are out of contract by design, matching IntelliJ's navigation semantics.

## History — whole-file degradation (FIXED, commit `23d1106`)

An earlier run flagged one dominant, actionable finding: **`ActorServiceImpl` whole-file resolver
degradation** — the file parsed, but *every* type mention safe-degraded to `UNRESOLVED`, so jcma
emitted nothing for the whole compilation unit (the lone owner type jcma never emitted). Root cause: a
`yield` (switch-expression arm) parsed at `LanguageLevel.RAW` produced a parse problem, so JavaParser
skipped `SymbolResolver` injection for the entire unit (and RAW's recovery discarded the yield arm).
Fixed by parsing at the project's discovered Java level with the language-level validator stripped
(docs/whole-file-resolution-degradation.md). Post-fix the never-emitted-owner count is 0.
